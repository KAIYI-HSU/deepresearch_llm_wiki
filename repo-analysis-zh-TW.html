01專案概述

LLM Wiki 是一款跨平台桌面應用程式，旨在解決傳統 RAG（Retrieval-Augmented Generation）系統的根本性缺陷： 知識不會累積。傳統 RAG 在每次查詢時從原始文件重新檢索與組合答案，這意味著昂貴的推理成本被重複支付， 且文件之間的洞見從不會「沉澱」下來。

LLM Wiki 的解決方案是讓 LLM 逐步建構並維護一個持久化、互相連結的 Markdown Wiki。 每當使用者匯入新文件，LLM 不只是把它索引起來，而是「閱讀 → 摘要 → 整合 → 更新交叉引用 → 標記矛盾 → 寫入紀錄」。 這份 Wiki 隨著時間推進不斷豐富，且因為儲存格式為純 Markdown，可以直接用 Obsidian 開啟瀏覽。

定位

個人 / 研究 / 團隊知識庫構建工具，主打「LLM 維護、人類策展」的角色分工。

哲學基礎

源自 Andrej Karpathy 公開的 llm-wiki.md 抽象模式，本專案是其具體實作版本。

技術形態

Tauri 桌面 App + Chrome Extension + 本地 HTTP API（port 19827）+ 可選向量資料庫。

02核心理念：為什麼 Wiki 而非 RAG？

「Wiki 是一個持久、複利累積的成果。交叉引用已經就位、矛盾已被標註、綜合理解反映了你讀過的全部內容。 維護一個知識庫繁瑣的不是閱讀或思考，而是記帳──而 LLM 不會無聊。」 —— 改寫自 Karpathy 原文

關鍵差異對比

面向傳統 RAGLLM Wiki
知識成果每次查詢時臨時組裝持久化 Markdown 檔案，可被人類瀏覽
跨文件洞見難以累積，每次重新發現已預先寫入頁面，並透過 wikilink 互聯
矛盾偵測無Ingest 時主動標註，Lint 時系統化檢查
查詢成本每次都需 embedding + LLM 重組Wiki 已是現成知識，可直接讀取
使用者所有權黑盒向量索引純 Markdown，可用 git / Obsidian 操作
工程複雜度低較高（需設計 ingest / lint / link 整套邏輯）

三個核心操作（Operations）

  • Ingest（攝入） — 加入新來源，LLM 讀取、摘要、更新 entity / concept 頁、寫 log。
  • Query（查詢） — 對 Wiki 提問，LLM 搜尋相關頁、合成答案，且好答案可被存回 Wiki。
  • Lint（健檢） — 定期檢查矛盾、孤立頁、過時主張、缺少的概念頁等。

04主要功能特色（相對 Karpathy 原模式的擴充）

原模式僅是一份抽象設計文件，本專案在此之上實作了 18+ 項具體擴充：

① 兩階段思維鏈 Ingest

原模式為單步攝入；本專案拆為「分析」與「生成」兩次 LLM 呼叫，提升品質： 第一次抽取實體 / 概念 / 矛盾，第二次據此產生 Wiki 頁面、index、log、review 項。

② SHA256 增量快取

來源檔案內容於攝入前計算雜湊，未變動的檔案自動跳過，節省 token 與時間。

③ 持久化 Ingest 佇列

序列處理避免並發呼叫，佇列存於磁碟、重啟可續，失敗自動重試最多 3 次。

④ 多模態圖片攝入

從 PDF / PPTX / DOCX 抽出嵌入圖片，以 Vision LLM 產生事實性描述（caption）， 供圖片感知搜尋與 lightbox 預覽。

⑤ 4 訊號相關性模型

知識圖譜邊權重綜合：直接連結 ×3、來源重疊 ×4、Adamic-Adar ×1.5、型別親和 ×1.0。

⑥ Louvain 社群偵測

自動發掘知識叢集，含內聚度評分（<0.15 警告）、12 色配色、社群圖例。

⑦ Graph Insights

自動偵測「驚奇連結」、「孤立頁面」、「橋接節點」三類洞見， 可一鍵觸發 LLM 優化的 Deep Research。

⑧ 多階段檢索管線

Tokenized 搜尋 → 可選向量搜尋（LanceDB）→ 圖譜擴展 → token 預算分配。 實測加上向量搜尋後 recall 從 58.2% → 71.4%。

⑨ 多會話聊天

獨立 session、對話側邊欄、可配置歷史深度、引用清單可摺疊、可一鍵 Regenerate、答案可存回 Wiki。

⑩ Thinking 顯示

對於發送 <think> 的模型（DeepSeek、QwQ）以滾動 5 行樣式呈現， 生成完成後可摺疊。

⑪ KaTeX 數學渲染

支援 inline / block，並自動為赤裸的 \begin{aligned} 包上 $$，含 100+ Unicode fallback。

⑫ 非同步 Review 系統

LLM 攝入時標記「需要人類判斷」項目，動作受限為 Create Page / Deep Research / Skip， 預先生成搜尋查詢字串。

⑬ Deep Research

Tavily 多查詢網路搜尋 + 全文擷取（不截斷）→ LLM 合成 → 自動 ingest。 最多 3 個並發任務，附專屬側邊面板。

⑭ Chrome Web Clipper

Manifest V3、Mozilla Readability + Turndown，本機 HTTP API 通信， app 不在線時可離線預覽，每 3 秒輪詢處理。

⑮ 多格式解析

PDF / DOCX / PPTX / XLSX / 圖片 / 影音皆有專屬處理路徑，產出結構化 Markdown。

⑯ 串聯刪除

刪除來源時用 3 種方法（frontmatter sources、來源摘要頁名、區段引用）找關聯頁， 共享實體頁僅移除該來源、不整頁刪除，並清理失效 wikilink 與 index。

⑰ 跨平台一致性

22+ 檔共用 normalizePath()、Unicode-safe 字元切片防 CJK 檔名崩潰、 macOS close-to-hide、Windows/Linux 退出確認。

⑱ 多 LLM 提供者

OpenAI / Anthropic / Google / Ollama / Custom 各有 streaming 與 header 的差異化處理， 可獨立配置 embedding 端點。

05技術細節深入剖析

5.1 兩階段 Ingest 管線

Step 1（分析）：LLM 讀取來源 → 結構化分析
  - 關鍵實體、概念、論點
  - 與既有 Wiki 內容的連結
  - 與既有知識的矛盾與張力
  - 對 Wiki 結構的建議

Step 2（生成）：LLM 接收分析 → 產生 Wiki 檔案
  - 來源摘要頁（含 frontmatter: type, title, sources[]）
  - Entity / Concept 頁與交叉引用
  - 更新 index.md, log.md, overview.md
  - Review 項目（給人類判斷）
  - Deep Research 用搜尋查詢字串

程式碼位於 src/lib/ingest.ts（1614 行）。設計上將「理解」與「寫作」拆開， 避免單次呼叫同時兼顧分析品質與輸出格式所造成的劣化，並讓中間結果易於除錯與快取。

5.2 4 訊號相關性模型（Knowledge Graph）

訊號權重意義
Direct link（直接 wikilink）×3.0頁面間透過 [[wikilink]] 直接連結
Source overlap（來源重疊）×4.0共享相同 raw 來源（最強訊號 — 共出處最能反映關係）
Adamic-Adar×1.5共同鄰居數，按鄰居度數倒數加權
Type affinity（型別親和）×1.0同型別頁加成（entity↔entity 等）

位於 src/lib/graph-relevance.ts（312 行）。視覺化使用 sigma.js + ForceAtlas2， 具備位置快取避免資料變動時版面跳動。

5.3 多階段檢索管線

Phase 1：Tokenized 搜尋
  - 英文：分詞 + 停用字
  - 中文：CJK bigram（每个 → [每个, 个…]）
  - 標題命中加成 +10
  - 同時搜尋 wiki/ 與 raw/sources/

Phase 1.5：向量語意搜尋（可選）
  - 透過任意 OpenAI 相容 /v1/embeddings 端點
  - 儲存於 LanceDB（Rust 後端）做 ANN 檢索
  - 用餘弦相似度，可發現關鍵字無交集但語意相關的頁
  - 結果與 tokenized 搜尋合併（已存在加成、新發現添加）

Phase 2：圖譜擴展
  - 取頂端搜尋結果為種子
  - 用 4 訊號相關性找關聯頁
  - 2-hop 衰減

Phase 3：預算控制
  - context window 4K → 1M tokens 可調
  - 比例分配：60% wiki / 20% chat 歷史 / 5% index / 15% system
  - 頁面以 search + graph 綜合分排序

Phase 4：上下文組裝
  - 編號頁全文（不只摘要）
  - System prompt 含 purpose.md / 語言規則 / 引用格式 / index.md
  - 指示 LLM 以 [1], [2] 形式引用

5.4 Rust 後端關鍵模組

  • src-tauri/src/commands/fs.rs — 統一的 preprocess_file：依副檔名分派到 PDF / DOCX / PPTX / XLSX 解析。
  • src-tauri/src/commands/extract_images.rs — 利用 pdfium page.objects() 抽取 PdfPageObjectType::Image，PPTX/DOCX 直接讀 zip 內 media/。
  • src-tauri/src/commands/vectorstore.rs — LanceDB 介面（vector_upsert_chunks 等）。
  • src-tauri/src/commands/claude_cli.rs — 用 tokio::process 啟動 claude 子行程，stdout 逐行串回前端。
  • src-tauri/src/clip_server.rs — tiny_http 起 port 19827，接 Chrome Extension 的剪輯 POST。
  • src-tauri/src/panic_guard.rs — 第三方解析器發生 panic 時於 Tauri command 邊界轉成 Result<Error>，避免單檔毀掉整個 App（搭配 panic = "unwind"）。

5.5 測試策略

專案測試覆蓋面相當紮實，從目錄結構可見：

  • 單元測試：*.test.ts（搜尋、tokenize、frontmatter、wikilink、merge、cleanup 等核心邏輯）
  • Property-based 測試：*.property.test.ts 用 fast-check 自動產生輸入（detect-language、path-utils、review-utils、sweep-reviews）
  • 情境測試：*.scenarios.test.ts（lint、ingest、enrich-wikilinks、search、sweep）
  • 真實 LLM 測試：*.real-llm.test.ts 透過 npm run test:llm 單獨跑，--no-file-parallelism 避免速率限制
  • 整合測試：*.integration.test.ts（ingest-queue、persist）
  • 競態測試：sweep-reviews.race.test.ts

5.6 跨平台與穩健性處理

  • 路徑統一：normalizePath() 在 22+ 檔被使用，全部反斜線→正斜線
  • Unicode 安全：字元級切片避免 CJK 檔名因 byte boundary 切到一半而崩潰
  • 專案互斥鎖：project-mutex.ts 防止多個視窗同時操作同一 Wiki
  • 15 分鐘 timeout：長 ingest 不會被預設逾時殺掉
  • dataVersion 訊號：Wiki 內容變動時自動觸發 graph 與 UI 重新整理
  • Release 設定：codegen-units = 1、lto = true、opt-level = "s"、panic = "unwind" 兼顧大小、速度與容錯

5.7.2 階段細節剖析

階段 ① 圖譜結構分析（觸發點）

與市面上 Deep Research 不同，這裡的觸發來源不是使用者的疑問，而是 graph-insights.ts 對知識圖譜的拓樸分析結果。系統會列出三類 Gap：

類別條件建議
孤立節點 (isolated-node) linkCount ≤ 1，排除 overview/index/log 「這些頁面缺乏連結，考慮補 wikilink 或研究擴充」
稀疏社群 (sparse-community) Louvain 社群 cohesion < 0.15 且 ≥ 3 節點 「此知識區塊內部交叉引用稀薄，研究以補洞」
橋接節點 (bridge-node) 單一節點連通 ≥ 3 個不同社群 「關鍵連接點若內容單薄會削弱整個 Wiki」

這是個關鍵設計：研究主題的「發現」交給圖論演算法而非使用者， 讓 LLM 主動指出「你的知識網路哪裡有破洞」。

階段 ② 主題優化（context-aware）

使用者點 Deep Research 後，並非直接搜尋圖譜給的 raw description，而是先請 LLM 將其精煉。 這是optimize-research-topic.ts 的工作： 

輸入：gapTitle + gapDescription + gapType
      + purpose.md（Wiki 為何而生）
      + overview.md（Wiki 目前涵蓋什麼）

輸出（嚴格 4 行格式）：
  TOPIC: <一句精準研究問題，使用使用者語言>
  QUERY: <查詢 1，可用英文關鍵字以符合搜尋引擎>
  QUERY: <查詢 2>
  QUERY: <查詢 3>

重點：topic 與 query 語言可不同──topic 跟隨使用者語言（中文 wiki 用中文 topic）， 但 query 可使用英文關鍵字以最大化搜尋引擎 recall。這個設計反映了實務上中文用戶常見的痛點： AI 直接用中文搜尋常會搜不到英文技術內容。

階段 ③ 可編輯確認對話框

優化後的 topic + queries 會塞進可編輯的 dialog（不是直接執行）。 使用者可微調 topic 文字、調整 / 刪除 / 替換 3 個查詢。 這是「人在迴路」（HITL）的關鍵閘門，避免 LLM 把研究方向理解錯後直接燒掉 API 額度。

階段 ④ 佇列（concurrent = 3）

任務經 queueResearch() 進入 Zustand store。maxConcurrent = 3 硬編碼上限──不是隨意決定，是因為下游 ⑥ 的 LLM 合成每個任務都會吃一個 stream connection， 多開易踩到 provider rate limit。狀態機如下： 

queued → searching → synthesizing → saving → done
                                              ↘ error

階段 ⑤ 多查詢網路搜尋

對每個 query 呼叫 tavilySearch()（或 SerpApi），各取 5 筆，以 URL 去重合併。 Tavily 端點呼叫使用 search_depth: "advanced" 取得較深入內容、include_answer: false 避免 Tavily 自己先給一個簡答（因為合成工作要交給後續 LLM）。

特別注意：HTTP 走 Tauri HTTP plugin 而非瀏覽器 fetch，避免 CORS 限制； 且為了將來換 Serper/Exa/Brave 不必各自處理跨域問題。

階段 ⑥ LLM 合成（含 wiki 交叉引用）

這是整個流程最關鍵的環節。system prompt 不僅要求合成，更強制要求：

  • 讀取 wiki/index.md 注入 prompt，列出既有 Wiki 所有頁面標題
  • 遇到既有實體必須用 [[wikilink]]──這是知識能否「接回」既有圖譜的關鍵
  • 網路來源以 [N] 編號引用
  • 標註矛盾、缺口、值得追加的來源
  • 中立、百科式語氣

輸出採用 SSE streaming，每個 token 進來就updateTask({ synthesis: accumulated })， UI 可即時看到內容生成。

階段 ⑦ 寫入 Wiki

檔名規則：research-<slug>-<YYYY-MM-DD>.md，存於 wiki/queries/。 寫入前先 strip 掉 <think>...</think> 區塊（含未閉合的）以保持頁面整潔。 Frontmatter 含 origin: deep-research 標籤，未來可以據此查詢 / 過濾 / 分析。

階段 ⑧ Auto-Ingest（複利累積）

這是本流程與業界 Deep Research 最大的差異：研究結果不是「給使用者看完就結束」， 而是被當作新來源、再次餵進兩階段攝入管線。 

autoIngest(projectPath, savedPath, llmConfig)
  ├ Step 1：分析 → 抽取此研究產出的實體 / 概念 / 矛盾
  └ Step 2：生成 → 建立 / 更新 entity, concept 頁
                  → 更新 index.md, log.md, overview.md
                  → wikilink 自動串接回既有頁

亦即：一次 Deep Research 至少觸發 3 次 LLM 呼叫（topic 優化 1 + 合成 1 + ingest 至少 2 = 通常 4 次）， 換來的是研究結果直接成為知識圖譜的新節點，並可能在下一輪 Insights 分析時又揭露新 Gap，形成自我強化的迴圈。

5.7.7 三個值得注意的工程細節

① 結構性頁面被排除在 Insights 之外

graph-insights.ts 明確將 { "index", "log", "overview" } 列為 STRUCTURAL_IDS 排除。 這些頁面與所有頁都連結，若納入分析會主導 surprising connections，掩蓋真正有趣的訊號。 這是個小但重要的工程經驗。

② Multi-query 是線性 for-loop 而非並發

在 deep-research.ts 中，3 個 query 是用 for...of 串列呼叫 tavilySearch。 看似不夠優化，實則合理：Tavily/SerpApi 都有 RPS 限制，串列呼叫穩定性勝過 Promise.all， 且整體仍在數秒級。若改並發要加 retry-with-backoff，工程成本不划算。

③ Auto-Ingest 用 .catch() 而非 await

最後一步 autoIngest(...).catch(err => console.error(...)) 是火後不理── 任務狀態 done 在寫入 wiki 頁面後立即標記，不等 ingest 完成。 這保證了 UI 即時回應，但代價是若 ingest 失敗，使用者只能在 Activity Panel 看到錯誤， 研究頁仍會留下、entity/concept 頁卻沒生成。是個合理的 UX/正確性 trade-off。

5.7.8 結論：本質上的範式差異

LLM Wiki 的 Deep Research 不是「OpenAI DR 的廉價版」，而是不同的範式：

  • 業界 DR 是會話內的長篇報告生成器──回答一個具體問題，產出一份完整文件
  • 本專案 DR 是圖譜持續演化的維護動作──修補一個結構缺陷，把結果焊回知識網路

其犧牲了單次研究的深度與廣度，換來了：

  1. 觸發自動化：研究主題不靠使用者靈感，靠演算法
  2. 結果持久化：研究產出不停留在對話，直接改變知識圖譜結構
  3. 複利累積：每次研究都可能在下一輪 Insights 中再被發現新缺口
  4. 成本低廉：可大量、頻繁使用，而非「重大研究任務」才動用

這個設計極度切合 LLM Wiki「個人長期知識複利」的核心定位。如果你期待 ChatGPT Deep Research 等級的單次報告品質， 本專案無法達成；但若你接受「每天小步、長期累積」，本專案的迴路反而比 agent loop 更貼合真實知識工作的節奏。

5.8Retrieval & Citation 完整生命週期

上節聚焦 Deep Research 的流程，本節進一步剖析其檢索（retrieval）與引用管理（citation management） 的內部機制。這部分是整個專案最被低估的設計細節──Deep Research 之所以能讓研究成果正確接回 Wiki 圖譜， 關鍵就在這套「雙軌檢索 + 雙軌引用 + 後置串接」的協作。

5.8.1 雙軌檢索（Dual-Track Retrieval）

Deep Research 的檢索不走主聊天的 RRF 管線，而是另設一條極簡的雙軌設計：

軌來源用途實作位置
外部軌 Tavily / SerpApi (3 queries × 5 results) 合成正文的主要事實來源 deep-research.ts:76-91 + web-search.ts
內部軌 僅 wiki/index.md 提供既有頁面清單，讓 LLM 知道哪些 entity 該打 wikilink deep-research.ts:107-112

關鍵設計決策：為什麼不讀內部頁面全文？

主聊天會 Phase 1-4 把相關頁面內容塞進上下文。Deep Research 刻意只讀 index.md，原因有三：

  1. 避免污染外部研究結論──若塞入內部頁面，LLM 可能會「重述既有知識」而非「整合新資訊」
  2. 節省 token──研究是新內容，內部全文是冗餘上下文
  3. 只需「標題清單」即可建立 wikilink 連結──知道有 anthropic.md 就能寫 [[anthropic]]

5.8.2 多查詢策略：3 × 5 × URL Dedup

// deep-research.ts:73-91
const allResults: WebSearchResult[] = []
const seenUrls = new Set<string>()

for (const query of queries) {  // 串列、非並發
  try {
    const results = await webSearch(query, searchConfig, 5)
    for (const r of results) {
      if (!seenUrls.has(r.url)) {
        seenUrls.add(r.url)
        allResults.push(r)
      }
    }
  } catch {
    // continue with other queries — 單 query 失敗不擋整體
  }
}

三個關鍵抉擇：

  • 固定 3 query × 5 result：上限 15 筆，去重後通常 8–12 筆，能控制 LLM context size
  • 串列而非並發：Tavily/SerpApi RPS 限制，串列更穩、整體仍在數秒級
  • URL 而非 hash 去重：簡單但有效；同一篇文章從不同搜尋引擎來可能有不同 query string，這裡用 raw URL 比對

5.8.3 Tavily API 的細節參數

// web-search.ts:90-100
await httpFetch("https://api.tavily.com/search", {
  method: "POST",
  body: JSON.stringify({
    api_key: apiKey,
    query,
    max_results: maxResults,
    search_depth: "advanced",     // 深度抓取全文，非 snippet
    include_answer: false,        // 不要 Tavily 自己合成答案
  }),
})

search_depth: "advanced" 是關鍵 —— 它讓 Tavily 回傳完整正文（截斷較少）， 這是後續 LLM 合成品質的基礎。include_answer: false 則明確拒絕 Tavily 的「快速答案」， 因為合成這項工作要交給後面的 LLM 配合 wiki 上下文一起做。

5.8.4 雙軌引用（Dual-Track Citation）

引用格式分兩種，由 system prompt 強制：

引用類型格式指向後續處理
外部來源 [N]（N 為編號） 網路文章 寫入頁尾 ## References 章節

System prompt 中的明確指令：

"## Cross-referencing (IMPORTANT)"
"- The wiki already has existing pages listed in the Wiki Index below."
"- When your synthesis mentions an entity or concept that exists in the wiki,"
"  ALWAYS use [[wikilink]] syntax to link to it."
"- For example, if the wiki has an entity 'anthropic', write [[anthropic]]"
"  when mentioning it."
"- This is critical for connecting new research to existing knowledge in the graph."

"## Writing Rules"
"- Organize into clear sections with headings"
"- Cite web sources using [N] notation"
"- Note contradictions or gaps"
"- Suggest additional sources worth finding"
"- Neutral, encyclopedic tone"

合成完成後，References 章節由程式碼自動產生，並非交由 LLM 自己整理：

const references = webResults
  .map((r, i) => `${i + 1}. [${r.title}](${r.url}) — ${r.source}`)

"type: query"
`title: "Research: ${topic}"`
"created: 2026-05-11"
"origin: deep-research"

"---"
""
`# Research: ${topic}`
""

5.8.6 引用清理：去除 <think> 區塊

寫入磁碟前還有一道清理：

// deep-research.ts:178-181
const cleanedSynthesis = accumulated
  .replace(/<think(?:ing)?>\s*[\s\S]*?<\/think(?:ing)?>\s*/gi, "")  // 閉合的
  .replace(/<think(?:ing)?>\s*[\s\S]*$/gi, "")                    // 未閉合的
  .trimStart()

這對 DeepSeek、QwQ 等會輸出推理過程的模型必要。 UI 顯示時可以摺疊展示 thinking，但儲存到 wiki 時不該污染最終內容。 兩個 regex 分別處理「閉合」與「stream 中途意外結束」情形。

Context 預算 依 maxCtx 比例分配（5/50/30/15） 無預算（合成是新內容主體）
Citation 格式 [1]..[N] + 隱藏 <!-- cited: 1,3,5 --> [N] 外部 + [[wikilink]] 內部
Citation 追蹤 extractCitedPages() → UI 引用卡片 ## References 寫入 markdown
結果持久化 聊天訊息（除非 Save to Wiki） 永久 Wiki 頁 + 自動 ingest 出新節點
引用追溯到圖譜 無（聊天訊息不在圖譜內） 透過 sources: [] frontmatter 永久串接

5.8.9 設計權衡與已知限制

① LLM 可能誤判 wikilink 目標

LLM 看到 index.md 中有 anthropic，可能會在不該打的場景也打 [[anthropic]] （例如另一家公司也叫 anthropic 的虛構場景）。當前沒有後驗證機制。

② 不檢測 dead wikilink

若 LLM 編造 [[non-existent-page]]，目前流程不會在合成階段檢出， 要等到後續 Lint 才會發現。原因：合成時不知道哪些頁面真實存在（只看 index.md）。

③ Citation 不去重

若 LLM 連續用 [1][1]，References 章節仍然只列一次（程式產生時是 webResults 一對一映射）， 但若 LLM 跳號（用了 [5] 但沒用 [1]–[4]），References 仍會列所有 5 筆 —— 預期是「列出全部，讓使用者自己看」。

④ Source overlap 計算延遲

圖譜邊「source overlap」依賴 autoIngest 完成；若 autoIngest 失敗（火後不理）， 研究頁就成了孤立節點，需要使用者手動重 ingest。

# Task: Reproduce LLM-Wiki's Deep Research Retrieval & Citation System

You are tasked with implementing a Deep Research pipeline that retrieves
information from the web AND cross-references it against an existing personal
wiki, producing a markdown research page with structured citations. The output
must be ingestible by a separate ingest pipeline to extract entities/concepts
into the wiki graph.

## High-Level Goals

1. Given a user-confirmed research topic + 3 search queries, run multi-query
   web search and synthesize results into a wiki-compatible markdown page.
2. Produce TWO kinds of citations:
   - `[N]` numeric brackets pointing to external web sources
   - `[[wikilink]]` syntax pointing to existing wiki entities/concepts
3. Make the result auto-ingestible — the wikilinks must form valid graph
   edges when a downstream ingest pipeline processes the page.

## Interface Contract

```typescript
interface ResearchInput {
  projectPath: string;           // root of the wiki project on disk
  topic: string;                 // user-confirmed research topic
  searchQueries: string[];       // 3 queries (length is conventional, not enforced)
  llmConfig: LlmConfig;          // streaming LLM client config
  searchConfig: SearchApiConfig; // Tavily or SerpApi credentials
}

interface WebSearchResult {
  title: string;
  url: string;
  snippet: string;  // FULL content if provider supports it
  source: string;   // hostname, e.g. "arxiv.org"
}

interface ResearchTask {
  id: string;
  topic: string;
  searchQueries: string[];
  status: "queued" | "searching" | "synthesizing" | "saving" | "done" | "error";
  webResults: WebSearchResult[];
  synthesis: string;          // accumulated LLM output
  savedPath: string | null;   // relative to projectPath
  error: string | null;
}
```

## Stage 1: Multi-Query Web Search (Dual-Track Retrieval)

```typescript
async function executeRetrieval(
  queries: string[],
  searchConfig: SearchApiConfig,
): Promise<WebSearchResult[]> {
  const allResults: WebSearchResult[] = [];
  const seenUrls = new Set<string>();

  // DESIGN: serial, not parallel. Search providers rate-limit; serial keeps
  // it simple and total time stays in seconds. DO NOT use Promise.all.
  for (const query of queries) {
    try {
      const results = await webSearch(query, searchConfig, 5);
      for (const r of results) {
        if (!seenUrls.has(r.url)) {
          seenUrls.add(r.url);
          allResults.push(r);
        }
      }
    } catch {
      // DESIGN: tolerate single-query failure. Continue with remaining.
    }
  }
  return allResults;
}

// Tavily call MUST use these parameters:
//   search_depth: "advanced"   — full content, not snippets
//   include_answer: false      — we synthesize ourselves, not Tavily
//   max_results: 5             — per query
```

## Stage 2: Internal Lightweight Retrieval

```typescript
// DESIGN: read ONLY wiki/index.md. Do NOT read full page content. Rationale:
//   1. We want NEW synthesis from external sources, not echoes of existing wiki.
//   2. Saves tokens — internal full text is irrelevant to the new content.
//   3. We only need page TITLES so the LLM knows what [[wikilinks]] are valid.
const wikiIndex = await readFile(`${projectPath}/wiki/index.md`).catch(() => "");
```

## Concurrency & Queue

- Maintain a task queue with state machine:
    queued → searching → synthesizing → saving → done | error
- Hard limit: maxConcurrent = 3. More than this triggers provider rate
  limits on most LLM backends.
- Tasks beyond the limit wait in `queued`; pick the next when one finishes.

## Mandatory Tests

1. **Multi-query dedup**: 3 queries returning overlapping URLs collapse to a
   single result set. Verify via Set size.
2. **Tavily failure tolerance**: One of three queries throws → other two
   still produce results and synthesis runs.
3. **Empty results path**: All queries return zero results → task transitions
   to `done` with synthesis = "No web results found." Auto-ingest skipped.
4. **<think> block stripping**: Synthesis containing `<think>...</think>`
   blocks (closed and unclosed) is cleaned before save.
5. **Frontmatter quote escaping**: A topic containing `"` characters
   produces valid YAML frontmatter.
6. **Slug edge cases**: Topic with unicode, emojis, very long text → slug
   stays ≤50 chars and only `[a-z0-9-]`.
7. **Wikilink survival**: A synthesis containing `[[anthropic]]` is
   preserved verbatim through save.
8. **References numbering matches**: If webResults has N items, References
   section has exactly N numbered lines.

## Non-Goals (Do NOT implement)

- DO NOT add agentic iteration (no plan→search→reflect→retry loop).
- DO NOT validate that wikilinks point to existing pages at synthesis time.
  Lint phase (separate pipeline) handles that.
- DO NOT deduplicate citations in the synthesis — `[1][1]` stays as written.
- DO NOT generate the References section with the LLM. Build it from code.
- DO NOT read internal wiki page bodies — only `wiki/index.md`.

5.8.11 結論

LLM Wiki 的 Retrieval & Citation 設計核心是「分離關注點，讓 LLM 只做它擅長的」：

  • 檢索由程式控制（多 query、URL 去重、固定上限）—— 確定性、可重現
  • 合成由 LLM 負責 —— 創造性整合，使用 [N] 與 [[wikilink]]
  • 引用組裝由程式控制（References 章節從 webResults 自動產生）—— 防止幻覺 URL
  • 引用持久化由 autoIngest 透過 sources: [] frontmatter 完成 —— 永久串接到圖譜

這個分工讓系統同時具備創造力與可靠性：LLM 寫的部分有彈性， 但 URL 引用是 deterministic 的；wikilink 仰賴 LLM 判斷，但圖譜邊的最終計算 靠 4 訊號相關性模型而非 LLM 的主觀關聯。整套設計才能讓研究結果正確、永久地融入知識網路。

07缺點與潛在風險 CONS

LLM 成本與速度

兩階段攝入需要兩次 LLM 呼叫，加上 overview.md 每次自動更新、Vision caption、可選 embedding， 重度使用下 token 成本不可忽視。雖有 SHA256 快取，第一次 ingest 仍是純成本。

Wiki 漂移（drift）風險

LLM 寫的頁面隨時間累積有可能風格不一、命名不統一、概念合併不徹底。 雖有 Lint 機制，但仍依賴使用者主動觸發、判讀建議。

規模上限不明

原模式作者建議 ~100 來源、數百頁面這個量級。當頁面達數千時，index.md 本身會變大、 圖譜效能、Louvain 計算時間、token 預算分配都會受壓。專案未明確標明效能曲線。

錯誤的「事實」會被寫入

LLM 攝入若誤讀來源（如表格抽取錯位、PDF OCR 雜訊、圖片 caption 不準），錯誤會寫入 Wiki 並透過 wikilink 擴散。 要靠 review + lint 補救，但人類負擔仍存在。

並發限制

Ingest 佇列為序列處理（為了避免 race condition 與提供穩定的 wiki 寫入）， 面對大量文件批次匯入時很慢。Deep Research 雖有並發 3 個但仍是少量。

跨來源重複資訊整合困難

同一個實體被 5 篇文章提及時，要不要合併、如何 reconcile 衝突，這類「真正困難」的整合工作仍高度依賴 LLM 判斷與 prompt 工程。從程式碼可見有 page-merge 邏輯但複雜度天花板存在。

Chrome Extension 安全面

本機 HTTP API（port 19827）對 http://127.0.0.1 開放、無認證機制（從 manifest 看）。 對單機個人使用問題不大，但若多使用者環境（例如共用機器、或惡意本機程式）會有未授權剪輯的風險。

對 LLM 提供者高度耦合

雖有抽象，但 streaming 格式差異仍須各自處理。LLM API 介面演化（如 Anthropic 工具呼叫格式變更）會直接造成維護負擔。

離線可用性受限

核心功能（ingest / query / research）皆依賴外部 LLM。雖支援 Ollama， 但本機模型品質與 token 上下文容量通常難以匹敵 GPT-4o / Claude Opus / Gemini，產出 wiki 品質會明顯下降。

學習曲線

使用者需理解 purpose.md / schema.md / index.md / log.md 各自角色才能發揮最大價值。 18+ 項功能（review、deep research、graph insights、lint…）對初學者門檻偏高。

GPL-3.0 授權限制

採 GPL-3.0，商業環境內嵌或衍生作品需 open source。對企業內部工具可接受， 但對 SaaS 嵌入或商業產品需謹慎評估法律風險。

缺乏多人協作

專案模型為「個人 / 單機」。雖然 wiki 是 git 友善的 markdown，但 review queue、ingest queue、 聊天紀錄都儲存在本機 .llm-wiki/，多人協作需自行設計同步機制。
